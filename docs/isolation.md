# Уровни изоляции транзакций

Документ объясняет, какие аномалии существуют, какие из них реально возможны
в нашем боте, и где мы используем какой уровень изоляции (и почему).

## Краткая таблица уровней (PostgreSQL)

| Уровень            | Dirty Read | Non-repeatable Read | Phantom Read | Write Skew      |
|--------------------|------------|---------------------|--------------|-----------------|
| READ UNCOMMITTED   | возможен¹  | возможен            | возможен     | возможен        |
| READ COMMITTED ⭐   | нет        | возможен            | возможен     | возможен        |
| REPEATABLE READ    | нет        | нет                 | нет²         | возможен        |
| SERIALIZABLE       | нет        | нет                 | нет          | **нет**         |

⭐ — дефолт в PostgreSQL и в нашем боте (через `DatabaseMiddleware`).
¹ — в PostgreSQL `READ UNCOMMITTED` фактически работает как `READ COMMITTED`.
² — REPEATABLE READ в PostgreSQL даёт snapshot isolation: phantoms тоже не видны.

## Что такое Write Skew (наш кейс)

Две транзакции читают **разные** данные, каждая проверяет инвариант локально,
и обе пишут. Результат: глобальный инвариант нарушается, хотя каждая
транзакция в отдельности корректна.

Классический пример в нашем боте — взаимный лайк:

```
Время   T1 (Alice → Bob)              T2 (Bob → Alice)
─────   ─────────────────              ──────────────────
t=0     BEGIN                          BEGIN
t=1     INSERT Like(A→B)
t=2                                    INSERT Like(B→A)
t=3     SELECT mutual WHERE B→A?
        → NULL (T2 не закоммитил)
t=4                                    SELECT mutual WHERE A→B?
                                       → NULL (T1 не закоммитил)
t=5     mutual is NULL, не создаём
        Match
t=6                                    mutual is NULL, не создаём
                                       Match
t=7     COMMIT                         COMMIT
─────────────────────────────────────────────────────────
Результат: 2 Like'а, 0 Match'ей. БАГ.
```

Под `READ COMMITTED` Postgres разрешает это, потому что чтение в `t=3`
действительно не "грязное" (T2 ещё не закоммитил) и каждая транзакция
сама по себе целостна. Аномалия только на уровне набора транзакций.

## Что используем где

### `bot/services/isolation.py:run_serializable`

Хелпер запускает блок кода в SERIALIZABLE-транзакции с retry на
SQLSTATE `40001` (`serialization_failure`). При конфликте Postgres
сам обнаруживает аномалию и аборти́рует одну из транзакций — мы её
ретраи́м с экспоненциальным backoff.

### `bot/handlers/browse.py:_persist_like_and_match` (SERIALIZABLE)

Критическая секция при свайпе вправо:
1. INSERT Like(from→to)
2. SELECT Like(to→from) — взаимный?
3. Если да — INSERT Match

Эти три операции делаются в одной SERIALIZABLE-транзакции через
`run_serializable`. Если две параллельные транзакции лайкают друг друга,
одна из них получит `40001` при коммите, мы её ретраи́м, и на ретрае она
видит закоммиченный лайк партнёра → создаёт Match.

Уведомления Telegram и публикация Celery-события вынесены за пределы
SERIALIZABLE-блока — они не влияют на инвариант и не должны держать
строгую блокировку.

### `bot/handlers/photos.py:cb_photo_up/_down` (SELECT FOR UPDATE)

Когда пользователь меняет порядок фото, мы свапа́ем `sort_order`. Если
два запроса прилетят почти одновременно (двойной тап), без блокировки
может возникнуть гонка: оба прочитают одни и те же `sort_order`,
оба попытаются записать → unique-constraint violation.

Используем **пессимистическую блокировку** — `SELECT … FOR UPDATE` в
**каноническом порядке id**. Канонический порядок предотвращает
deadlock'и (два встречных reorder'а не схватят локи в перекрёстном
порядке).

Здесь SERIALIZABLE был бы избыточен — конфликт чисто локальный
(одна и та же таблица, известные id), пессимистическая блокировка
эффективнее.

### Всё остальное (READ COMMITTED)

Запросы профиля, выдачи фида, чтения мэтчей, фотографий — нет инвариантов
между сущностями, которые могли бы быть нарушены. READ COMMITTED достаточен
и даёт максимальную пропускную способность.

## Как это проверить

Интеграционные тесты в [tests/test_isolation.py](../tests/test_isolation.py)
демонстрируют оба сценария на **реальном Postgres**:

```powershell
# 1. Поднять Postgres (либо весь стек, либо только БД)
docker-compose up -d postgres

# 2. Указать URL для интеграционных тестов
$env:INTEGRATION_PG_URL = "postgresql+asyncpg://postgres:postgres@localhost:5432/dating_bot"

# 3. Запустить
python -m pytest tests/test_isolation.py -v
```

Тесты:
- `test_mutual_like_race_under_read_committed` — **доказывает** существование
  бага под `READ COMMITTED`: два параллельных лайка проходят, Match не
  создаётся. Тест синхронизирует транзакции через `asyncio.Barrier`,
  чтобы они гарантированно были "в полёте" одновременно.
- `test_mutual_like_serializable_creates_match` — **доказывает** что фикс
  работает: ровно один Match создаётся под SERIALIZABLE-логикой `_do_like`.

## Trade-offs

**Цена SERIALIZABLE**: при высокой конкуренции на одних и тех же строках
часть транзакций будет аборти́роваться и ретра́иться, увеличивая latency.
В нашем случае конкуренция возникает только когда два конкретных
пользователя лайкают друг друга строго одновременно — это **редкое**
событие, retry-цена ничтожна по сравнению с риском "потерянного" мэтча.

**Цена SELECT FOR UPDATE**: блокирует строку до конца транзакции, мешая
конкурентным чтениям/записям той же строки. Для reorder это локально и
кратко (миллисекунды), приемлемо.

**Цена READ COMMITTED везде остальное**: ноль. Это дефолт, который нам
подходит для подавляющего большинства операций.

## Дальнейшие шаги

- Если появится таблица `dialogs` / `messages` — счётчик непрочитанных
  сообщений будет ещё одним кандидатом на write-skew (UPDATE counter
  WHERE id = ?). Решается атомарным SQL (`UPDATE … SET cnt = cnt + 1`)
  или SELECT FOR UPDATE.
- Если планируется кэшировать выдачу `get_ranked_candidates` агрессивнее —
  стоит подумать про read-your-writes consistency между Redis и Postgres.
