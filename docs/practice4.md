# Практика 4 — Аномалии изоляции SQL

## Что здесь происходит и зачем

Когда несколько транзакций работают с БД одновременно, возникают **аномалии изоляции** — ситуации, когда каждая транзакция в отдельности корректна, но их параллельное выполнение даёт неожиданный результат.

Стандарт SQL определяет четыре таких аномалии и четыре уровня изоляции, которые их предотвращают. В этой практике мы воспроизводим все четыре аномалии на живых транзакциях.

## Почему именно эти 4 аномалии

Выбраны **все четыре** классические аномалии из стандарта SQL-92, потому что вместе они дают полную картину проблемы параллелизма:

| Аномалия | Что ломается |
|----------|-------------|
| Dirty Read | Читаем данные, которых ещё «не существует» (транзакция может откатиться) |
| Non-Repeatable Read | Один и тот же SELECT в транзакции даёт разные строки |
| Phantom Read | Один и тот же SELECT даёт разное количество строк |
| Lost Update | Два параллельных обновления → одно исчезает |

---

## Почему PostgreSQL, а не MySQL

PostgreSQL — промышленная СУБД с хорошим соответствием стандарту. Она удобна тем, что:

1. Поддерживает все 4 уровня изоляции (хотя READ UNCOMMITTED = READ COMMITTED).
2. Все аномалии 2–4 реально воспроизводятся на уровне READ COMMITTED.
3. Уже использовалась в предыдущих практиках — меньше инфраструктуры.

Единственный нюанс: **Dirty Read** в PostgreSQL не воспроизводится — PG не реализует настоящий READ UNCOMMITTED (это задокументировано в официальной документации PG). Для демонстрации dirty read нужна MySQL с `SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED`. В нашем скрипте мы показываем *попытку* воспроизвести и фиксируем, что PG защищает от этой аномалии — что само по себе является важным результатом.

## Как объяснить преподавателю

### Dirty Read
> «Транзакция прочитала изменение, которое ещё не сохранено. Если другая транзакция откатится — у нас в памяти будет значение, которого никогда не существовало. PostgreSQL это предотвращает на уровне архитектуры (MVCC), но MySQL позволяет через READ UNCOMMITTED.»

### Non-Repeatable Read
> «Я начал транзакцию, прочитал баланс = 1000, пошёл считать. Пока считал — другая транзакция изменила строку. Читаю снова — уже 1500. Один SELECT, одна транзакция, два разных ответа. Решение — REPEATABLE READ: транзакция "заморожена" на снимке данных.»

### Phantom Read
> «Я посчитал заказы дороже 100 рублей — их 2. Пока я работал, кто-то вставил третий. Снова считаю — уже 3. Строка "появилась из ниоткуда" (фантом). Non-Repeatable Read про изменение строки, Phantom Read — про появление/исчезновение строк. Решение — SERIALIZABLE.»

### Lost Update
> «Два кассира одновременно читают баланс 1000₽ и начисляют разные суммы. Оба записывают "1000 + X". Кто последний — тот и победил, первое обновление потеряно. Решение: либо атомарный `UPDATE balance = balance + X` (без предварительного чтения), либо `SELECT ... FOR UPDATE` чтобы заблокировать строку.»

## Уровни изоляции и аномалии (шпаргалка)

```
READ UNCOMMITTED   → все аномалии возможны
READ COMMITTED     → dirty read закрыт, остальные возможны   ← PostgreSQL по умолчанию
REPEATABLE READ    → dirty + non-repeatable закрыты
SERIALIZABLE       → все закрыты, но самый медленный
```

Повышение уровня изоляции = больше блокировок или конфликтов = ниже производительность.  
Поэтому по умолчанию большинство СУБД использует READ COMMITTED, а не SERIALIZABLE.
