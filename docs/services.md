# Services & Responsibilities (Stage 1)

В этом документе фиксируем минимально-реалистичную архитектуру Dating-бота для защиты на Этапе 1.
Стек на Этап 1: `Python` + `aiogram`, БД `PostgreSQL`, фоновые задачи (в перспективе) через `Celery`.

## 1) Bot Service

Отвечает за взаимодействие с Telegram и “операции, которые требуют ответа пользователю”.

Ключевые обязанности:
- Принимает апдейты Telegram (webhook или long polling).
- Реализует команды и сценарии:
  - `/start` (регистрация пользователя по `telegram_id`).
  - Заполнение профиля (базовые поля анкеты).
  - Управление анкетой (создание/обновление/удаление с минимальным CRUD).
  - Поток “просмотра анкет” (показ кандидатных анкет и обработка действий: like / skip).
- Хранит минимальное “состояние диалога” (например, на стороне БД в `dialogs`/`messages` или во внутреннем состоянии приложения).
- Запускает фоновые задачи, когда действие пользователя требует пересчета или подготовки данных:
  - обновление рейтинга пользователя
  - подготовка списка кандидатов для текущей “сессии”

Контракты (логические):
- На события действий пользователя создаются записи в БД (например, `like`, `match`), после чего Bot Service инициирует соответствующую фоновую задачу.

## 2) Worker Service

Отвечает за “тяжелую” или “отложенную” бизнес-логику, чтобы Bot Service отвечал быстро.

Рекомендуемые роли Worker Service на будущих этапах (часть из них можно задокументировать уже сейчас):
- Пересчет ранжирования (уровни рейтинга):
  - `level1` (по полям анкеты)
  - `level2` (по поведению: лайки, пропуски, взаимные лайки, частота диалогов и активность по времени)
  - `level3` (комбинация уровней + дополнительные факторы, например рефералы)
- Регулярные пересчеты и “догрев” данных:
  - подготовить предранжированный список анкет для сессии пользователя
  - обеспечить “следующую пачку” анкет, когда пользователь дошел до конца текущей выдачи

Контракты задач (примерные, без жесткой привязки к коду на Этап 1):
- `recalculate_user_rating(user_id)`
- `prefetch_session_candidates(user_id, session_id, limit)`

## 3) PostgreSQL (Data Store)

Единый источник данных для:
- пользователей и их профилей
- анкет и материалов (например, фото)
- взаимодействий (лайки, матчи)
- диалогов и сообщений (если требуется в рамках ТЗ)
- рейтингов (храним значения и/или события, из которых рейтинг восстанавливается)

Индексы/ограничения:
- Уникальность пользовательских взаимодействий (чтобы исключить дубли).
- Индексы по FK и часто используемым полям (`telegram_id`, `profile_id`, `created_at`, и т.п.).

## 4) Опциональные компоненты (фиксируем как “следующие шаги”)

Эти элементы не обязательно реализовывать на Этапе 1, но полезно явно указать в архитектурной документации как “куда расширяемся”:
- `Redis`:
  - кэширование предварительно отранжированных списков анкет
  - кэширование справочников, если они редко меняются
- `MQ Broker` (например RabbitMQ):
  - потоковая обработка событий взаимодействий между сервисами
- `S3-совместимое хранилище` (например MinIO):
  - хранение изображений пользователей
- Метрики и логирование:
  - сбор статистики по запросам, ошибкам, времени выполнения задач

## 5) Минимальный набор сценариев для документации

Чтобы схема и БД были “привязаны к реальному поведению”, в документах удобно зафиксировать сценарии:
- `/start` -> создание записи пользователя
- Заполнение профиля -> обновление `user_profiles`
- Добавление анкеты/фото -> создание `user_photos`
- Пользователь нажал `like`:
  - создание записи в `likes`
  - при взаимном `like` -> создание `matches`
  - запуск фонового пересчета `ratings` (или подготовка обновленного рейтинга)
- Пользователь дошел до конца текущей выдачи:
  - Bot service запрашивает следующие кандидаты (логически через DB/Worker)

## 6) Чем реализуется каждый микросервис (фиксируем стек)

Ниже — “как на практике это будет написано” (для защиты на Этапе 1 удобно иметь именно такую таблицу соответствия).

- `Bot Service` (Telegram)
  - `aiogram` (обработчики апдейтов)
  - `SQLAlchemy` (доступ к `PostgreSQL`)
  - `Alembic` (миграции по схеме БД, на Этапе 1 это описываем, на следующих — реализуем)

- `Profiles Service`
  - `FastAPI` (API-слой для CRUD анкеты/профиля)
  - `SQLAlchemy` + `PostgreSQL`

- `Interactions Service`
  - `FastAPI` (API-слой для событий `like/skip/match`)
  - `SQLAlchemy` + `PostgreSQL`
  - `RabbitMQ` (публикация событий взаимодействий; клиент уровня очередей в Python)

- `Dialogs Service`
  - `FastAPI` (создание диалогов и запись/чтение сообщений)
  - `SQLAlchemy` + `PostgreSQL`

- `Ranking Service`
  - Python-логика расчета рейтинга
  - `SQLAlchemy` + `PostgreSQL` (чтение входных данных и запись `ratings`)

- `Celery Workers`
  - `Celery` (фоновые задачи: пересчет рейтинга, подготовка кандидатов)
  - `RabbitMQ` (как транспорт/очередь задач)
  - `SQLAlchemy` + `PostgreSQL` (обновление результатов расчета)

- `Redis`
  - кэширование/прогрев выдачи (`prefetched candidates`), чтобы ускорить первый запрос пользователя в сессии

- `Media Service` + `MinIO (S3-compatible)`
  - `FastAPI` (обработка загрузок/управление ссылками)
  - клиент S3/MinIO в Python для записи файлов
  - `MinIO` как хранилище изображений
