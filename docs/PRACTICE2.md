# Практика 2 — RabbitMQ vs Redis: сравнение брокеров сообщений

## Что здесь происходит

Бенчмарк запускает одинаковую нагрузку через два брокера — **RabbitMQ** и **Redis** —
и собирает измеримые метрики: пропускную способность, задержку и потери сообщений.

Для каждой комбинации `(брокер × размер сообщения × целевая скорость)` запускается
отдельный тест длительностью 30 секунд.

### Что тестируется

| Параметр | Значения |
|---|---|
| Размер сообщения | 128 B · 1 KB · 10 KB · 100 KB |
| Целевая скорость | 1 000 · 5 000 · 10 000 msg/s |
| Брокеры | RabbitMQ 3.13 · Redis 7 |
| Длительность теста | 30 секунд на прогон |
| Всего прогонов | 24 (2 брокера × 4 размера × 3 скорости) |

---

## Архитектура

```
┌─────────────┐     сообщения     ┌──────────────┐     ack     ┌─────────────┐
│  Producer   │ ─────────────────▶│    Broker    │ ──────────▶ │  Consumer   │
│  (thread)   │                   │ RabbitMQ /   │             │  (thread)   │
└─────────────┘                   │    Redis     │             └─────────────┘
                                  └──────────────┘
```

Всё работает внутри одного Python-процесса (`benchmark.py`) в двух потоках — producer и consumer.
Каждый поток создаёт своё подключение к брокеру.

### RabbitMQ
- Producer → `basic_publish` в именованную очередь
- Consumer → `basic_consume` + `basic_ack`
- Транспорт: **AMQP**, библиотека `pika`

### Redis
- Producer → `XADD` в Redis Stream с ограничением `MAXLEN=5000`
- Consumer → `XREADGROUP` (consumer group) + `XACK`
- Транспорт: **RESP**, библиотека `redis-py`

### Как считается латентность
В каждое сообщение записывается `{"ts": <unix_timestamp>, "data": "..."}`.
Consumer при получении вычисляет `latency = (now - ts) * 1000` мс.
Первые 2 секунды прогона исключаются из статистики (прогрев).

### Ограничение скорости (Rate Limiter)
Класс `RateLimiter` работает по принципу токен-бакет на монотонных часах:
каждый `basic_publish` / `XADD` вызывает `rl.wait()`, которая досыпает ровно столько,
чтобы соблюсти целевую частоту.

---

## Метрики

| Метрика | Как считается |
|---|---|
| `Tput/s` | `received / duration` |
| `Avg ms` | среднее арифметическое всех задержек |
| `P95 ms` | 95-й перцентиль задержек |
| `Lost` | `sent − received` |

## Как запустить

### Вариант 1 — через Docker (рекомендуется)

```bash
cd practice/practice2

# Запустить всё (брокеры + бенчмарк)
docker compose up --build

# Только посмотреть вывод бенчмарка
docker compose logs -f benchmark
```

Бенчмарк стартует автоматически после того, как оба брокера пройдут healthcheck.
Полный прогон занимает **≈ 25 минут** (24 теста × ~30 с + накладные расходы).

Чтобы остановить раньше времени:
```bash
docker compose down
```

### Вариант 2 — локально (RabbitMQ и Redis уже запущены)

```bash
cd practice/practice2

pip install -r requirements.txt

# Указать адреса брокеров (если не localhost)
export RABBITMQ_HOST=localhost
export REDIS_HOST=localhost

python benchmark.py
```

## Как читать вывод

Во время прогона каждый тест печатает одну строку:

```

  → sent=150,000  recv=149,987  lost=13  tput=4999/s  avg=1.2ms  p95=3.4ms
```

В конце — итоговая таблица:

```
=============================================================================
BENCHMARK RESULTS
=============================================================================

------------ ------- --------- --------- --------- ------- --------- --------- ---------
rabbitmq     128B      1,000    30,000    30,000       0     1000     0.8       1.4
```

## Что означают результаты

### Деградация single instance

Если брокер не справляется с нагрузкой — вы увидите:

- **Lost > 0** — сообщения потеряны (очередь переполнена или триммингRaised MAXLEN)
- **Tput/s < Target/s** — реальная пропускная способность ниже целевой
- **P95 ms резко растёт** — очередь накапливается, latency tail уходит вверх

### Ожидаемые наблюдения

| Сценарий | Что обычно происходит |
|---|---|
| Маленькие сообщения (128 B), низкая нагрузка | Redis быстрее за счёт in-memory RESP-протокола |
| Большие сообщения (100 KB), высокая нагрузка | Оба деградируют; Redis раньше теряет сообщения из-за `MAXLEN` |
| 10 000 msg/s с любым размером | Момент, где single instance начинает накапливать backlog |
| Стабильная нагрузка 1 000 msg/s | Оба справляются, RabbitMQ показывает чуть бо́льшую латентность из-за AMQP-overhead |
