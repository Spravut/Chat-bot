# Отчёт: Сравнение RabbitMQ и Redis как брокеров сообщений

## Стенд и методология

**Инструмент:** собственный producer на Python (`benchmark.py`)  
**Брокеры:** RabbitMQ 3.13-management-alpine, Redis 7-alpine  
**Среда:** Docker Compose, все три контейнера на одной машине  
**Длительность каждого прогона:** 10 секунд (первые 2 с исключены из статистики latency как прогрев)

Условия одинаковые для обоих брокеров:
- один producer-тред и один consumer-тред
- одинаковый формат сообщений: `{"ts": <unix_timestamp>, "data": "<payload>"}`
- одинаковые размеры payload и целевые скорости
- без персистентности (transient queues / Redis Streams без AOF)

Метрики собирались параллельно в отдельном тред-мониторе (раз в секунду):
- **backlog** — глубина очереди: Management API для RabbitMQ, `XLEN` для Redis
- **память брокера** — `/api/nodes` для RabbitMQ, `INFO memory` для Redis

## Анализ по экспериментам

### 1. Базовое сравнение (128B, 1 000 msg/s)

Оба брокера справились без потерь. RabbitMQ: 999/s при avg=1.1 ms, Redis: 1000/s при avg=0.5 ms. Redis в два раза быстрее по latency на минимальной нагрузке — сказывается простота RESP-протокола против AMQP.

### 2. Влияние размера сообщения

RabbitMQ деградирует по throughput и latency с ростом payload, Redis держится стабильно вплоть до 100KB:

| Size  | RabbitMQ tput/s | RabbitMQ avg ms | Redis tput/s | Redis avg ms |
|-------|-----------------|-----------------|--------------|--------------|
| 128B  | 999–1,432       | 1.1–2.0         | 1,000–2,636  | 0.5–194.8    |
| 1KB   | 999–1,240       | 1.2–2.3         | 1,000–2,539  | 0.7–162.5    |
| 10KB  | 999–1,234       | 2.0–2.6         | 1,000–1,460  | 0.8–2.2      |
| 100KB | 745–820         | 3,424–3,660     | 1,000–1,169  | 2.8–3.4      |

**Вывод:** Redis выигрывает на всех размерах. Особенно показателен 100KB: Redis даёт avg=2.8–3.4 ms, RabbitMQ — avg=3,424–3,660 ms (в тысячу раз медленнее). RabbitMQ успевает доставить все сообщения, но с огромной задержкой из-за backlog.

### 3. Влияние интенсивности потока

Ни один брокер не достиг целевых 5 000 и 10 000 msg/s при одном producer-треде. Фактический потолок:
- **RabbitMQ** — ~1 000–1 432 msg/s независимо от target
- **Redis** — ~1 000–2 636 msg/s в зависимости от размера сообщения

Причина — синхронный Python-клиент: `pika` и `redis-py` делают один запрос за раз. Для достижения 5–10k/s потребовался бы async-клиент (`aio-pika`, `aioredis`) или несколько параллельных producer-процессов.

**Точка деградации RabbitMQ по latency:** на 100KB avg latency вырастает до 3.5 секунды при сохранении полной доставки (0 потерь). Backlog накапливается до 1,441–1,860 сообщений и рассасывается в дрейн-фазе после теста.

**Redis:** стабилен на всех режимах. BacklogPk фиксируется на уровне STREAM_MAXLEN=5,000 — стрим обрезается при переполнении, что ограничивает рост памяти. В 100KB прогонах Redis использует до 1,838 MB RAM.

## Выводы

### Какой брокер показал бо́льшую пропускную способность?

**Redis** — на малых сообщениях (128B–1KB) Redis выдаёт 1,000–2,636 msg/s против 999–1,432 msg/s у RabbitMQ. На 100KB Redis также впереди: 1,000–1,169 msg/s против 745–820 msg/s.

### Какой брокер лучше переносит рост размера сообщения?

**Redis** лучше справляется на всём диапазоне. Latency Redis при 100KB (2.8–3.4 ms) практически не отличается от latency на 128B (0.5–0.7 ms). RabbitMQ при 100KB уходит в avg latency 3.4–3.6 секунды — backlog накапливается быстрее, чем consumer его разгребает.

### При какой нагрузке начинают деградировать single instance?

| Брокер   | Первые симптомы               | Характер деградации                                   |
|----------|-------------------------------|-------------------------------------------------------|
| RabbitMQ | 10KB/5000/s (tput падает до 1,158/s) | Плавное снижение throughput, рост latency на 100KB до 3.5 с, 0 потерь |
| Redis    | 128B/5000/s (latency 194 ms)  | Latency скачет при перегрузке стрима, throughput держится, 0 потерь |

### Какой инструмент лучше подходит для такого сценария?

Собственный producer — единственный адекватный выбор для сравнения разнопротокольных брокеров. `k6` и `Locust` заточены под HTTP и не имеют нативных AMQP/Redis-Stream коннекторов. Собственный producer даёт полный контроль над форматом сообщений, rate limiting, измерением end-to-end latency через embedded timestamp и одинаковыми условиями для обоих брокеров.

Для достижения реальных 5,000–10,000 msg/s следующий шаг — перейти на async-клиенты (`aio-pika` для RabbitMQ, `redis.asyncio` для Redis) и запустить несколько параллельных producer-процессов.

### Итоговая рекомендация

| Сценарий | Рекомендация |
|---|---|
| Маленькие сообщения (<10KB), высокий throughput | Redis — в 2× быстрее по latency, выше throughput |
| Большие сообщения (>10KB), низкая latency критична | Redis — avg latency 3 ms против 3.5 с у RabbitMQ |
| Ограниченная RAM на сервере | RabbitMQ — потребляет в 10+ раз меньше памяти |
| Нужен гарантированный порядок и ack | RabbitMQ — AMQP обеспечивает это из коробки |
