# Отчёт: Сравнение RabbitMQ и Redis как брокеров сообщений

## Стенд и методология

**Инструмент:** собственный producer на Python (`benchmark.py`)  
**Брокеры:** RabbitMQ 3.13-management-alpine, Redis 7-alpine  
**Среда:** Docker Compose, все три контейнера на одной машине  
**Длительность каждого прогона:** 10 секунд (первые 2 с исключены из статистики latency как прогрев)

Условия одинаковые для обоих брокеров:
- один producer-тред и один consumer-тред
- одинаковый формат сообщений: `{"ts": <unix_timestamp>, "data": "<payload>"}`
- одинаковые размеры payload и целевые скорости
- без персистентности (transient queues / Redis Streams без AOF)
- **одинаковый размер буфера брокера: 1 000 сообщений** — Redis Stream `MAXLEN=1000`, RabbitMQ `x-max-length=1000` с политикой `drop-head`

Метрики собирались параллельно в отдельном тред-мониторе (раз в секунду):
- **backlog** — глубина очереди: Management API для RabbitMQ, `XLEN` для Redis
- **память брокера** — `/api/nodes` для RabbitMQ, `INFO memory` для Redis

## Анализ по экспериментам

### 1. Малые сообщения: 128B и 1KB

Оба брокера справляются без потерь на всех целевых скоростях. Разница:

- **Redis backlog = 1,000** (ровно STREAM_MAXLEN) — стрим постоянно заполнен, но consumer читает с той же скоростью что producer пишет. Потерь нет.
- **RabbitMQ backlog = 0–3** — очередь почти пустая, consumer разгребает мгновенно.

Redis выдаёт более высокий throughput: 1,607–1,663/s против 1,071–1,098/s у RabbitMQ при target=5,000–10,000/s. Причина — RESP-протокол проще AMQP: нет framing overhead, routing, exchange lookup.

### 2. Средние сообщения: 10KB

Оба брокера по-прежнему без потерь, но RabbitMQ начинает проигрывать по throughput:

| Broker   | Tput/s (target 10,000) | Avg ms |
|----------|------------------------|--------|
| RabbitMQ | 937                    | 3.4    |
| Redis    | 1,072                  | 3.0    |

RabbitMQ не дотягивает до 1,000/s при target=5,000–10,000/s. Redis держит 1,000–1,072/s. Backlog Redis = ровно 1,000: буфер полон, но баланс сохраняется.

### 3. Крупные сообщения: 100KB — broker-level дропы

Именно здесь проявляется принципиальное отличие.

**RabbitMQ:**
- Producer: ~670/s, Consumer: ~474/s → дисбаланс +196 msg/s
- Буфер на 1,000 сообщений заполняется за ~5 секунд
- Затем `drop-head` выбрасывает ~196 сообщений/сек прямо в брокере — consumer их никогда не видит
- За 10 секунд: ~1,960 дропов, что точно совпадает с наблюдаемыми 1,953–2,347
- **Потери: 28–34%**
- Avg latency 1,448–1,458 ms — высокая, потому что оставшиеся сообщения ждут в очереди у потолка

**Redis:**
- Producer: ~906–932/s, Consumer: ~906–932/s → полный баланс
- Backlog = 1,000 (стрим у MAXLEN), но consumer читает с той же скоростью
- **Потери: 0**
- Avg latency 4.8–6.1 ms

**Почему RabbitMQ теряет баланс на 100KB?**

AMQP — тяжёлый протокол. Фрейминг, routing через exchange, per-message ack на каждое сообщение — всё это дорожает пропорционально размеру payload. `pika` подтверждает каждое сообщение отдельным round-trip. Redis RESP проще: `XREADGROUP` возвращает батч, накладные расходы не зависят от размера так сильно. При 128B–10KB этот overhead незаметен. При 100KB consumer начинает отставать от producer — разрыв накапливается, буфер переполняется, broker начинает дропать.

## Выводы

### Какой брокер показал бо́льшую пропускную способность?

**Redis** на всём диапазоне размеров. На малых сообщениях (128B–1KB) Redis выдаёт 1,600–1,700 msg/s против 1,000–1,100 у RabbitMQ. На 100KB Redis сохраняет ~920/s, RabbitMQ деградирует до ~470/s с потерями.

### Какой брокер лучше переносит рост размера сообщения?

**Redis** — throughput и latency почти не меняются от 128B до 100KB (0.5→5 ms avg). RabbitMQ деградирует плавно до 10KB, а на 100KB переходит в режим broker-level дропов.

### При какой нагрузке начинают деградировать single instance?

| Брокер   | Первые симптомы                       | Характер деградации                                         |
|----------|---------------------------------------|-------------------------------------------------------------|
| RabbitMQ | 10KB: tput падает до ~937/s           | Плавное снижение throughput с ростом payload                |
| RabbitMQ | **100KB: broker-level дропы 28–34%**  | Producer опережает consumer → `drop-head` выбрасывает сообщения |
| Redis    | 128B/5,000: latency скачет до 29 ms   | Latency растёт при высокой нагрузке, throughput держится   |
| Redis    | 100KB: 0 потерь, latency 5–6 ms       | Баланс сохраняется, деградации нет                         |

### Какой инструмент лучше подходит для такого сценария?

Собственный producer — единственный адекватный выбор для сравнения разнопротокольных брокеров. `k6` и `Locust` заточены под HTTP и не имеют нативных AMQP/Redis-Stream коннекторов. Собственный producer даёт полный контроль над форматом сообщений, rate limiting, измерением end-to-end latency через embedded timestamp и одинаковыми условиями для обоих брокеров.

### Итоговая рекомендация

| Сценарий | Рекомендация |
|---|---|
| Малые сообщения (<10KB), высокий throughput | Redis — быстрее, ниже latency, протокол проще |
| Большие сообщения (>10KB), надёжность критична | Redis — не теряет сообщения даже на 100KB |
| Ограниченная RAM | RabbitMQ — потребляет в 10+ раз меньше памяти |
| Нужен гарантированный порядок, ack, routing | RabbitMQ — AMQP обеспечивает это из коробки |
| Высокая нагрузка + крупный payload | Redis — RabbitMQ теряет 28–34% сообщений на broker-level |
