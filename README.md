# Chat-bot (Dating App)

Dating-бот для Telegram: анкеты, лайки/матчи, диалоги и система ранжирования пользователей (уровни 1-3).

## Стек

| Технология | Роль |
|---|---|
| `aiogram 3` | Async-фреймворк для Telegram Bot API |
| `PostgreSQL` | Основная база данных |
| `SQLAlchemy 2 (async)` | ORM — Python-объекты вместо сырого SQL |
| `asyncpg` | Async-драйвер PostgreSQL (используется под капотом SQLAlchemy) |
| `Alembic` | Версионные миграции схемы БД |
| `Docker / docker-compose` | Запуск PostgreSQL + бота одной командой |

## Этап 2: Разработка базовой функциональности ✅

Telegram-бот с регистрацией через FSM и хранением профилей в PostgreSQL.

### Быстрый старт

```bash
# Вариант 1 — Docker (PostgreSQL + бот одной командой)
docker-compose up --build

# Вариант 2 — локально (нужен запущенный PostgreSQL)
pip install -r requirements.txt
alembic upgrade head   # применяем миграции к БД
python -m bot.main     # запускаем бота
```

### Флоу регистрации

```
/start
  ├─ новый пользователь
  │    └─ FSM: имя → возраст → пол → кого ищет → город → bio (можно пропустить)
  │         → сохранение User + UserProfile в PostgreSQL
  │         → показ готовой анкеты + главное меню
  │
  └─ уже зарегистрирован
       └─ приветствие по имени + главное меню
```

## Описание каждого файла

### `bot/config.py`

Читает переменные окружения из `.env` файла:
- `BOT_TOKEN` — токен от BotFather для доступа к Telegram API
- `DATABASE_URL` — строка подключения к PostgreSQL вида `postgresql+asyncpg://user:pass@host/db`

Используется во всех модулях, которым нужен токен или соединение с БД.

---

### `bot/main.py`

Точка входа. Здесь собирается всё приложение:

1. Создаётся объект `Bot` с токеном — он умеет отправлять сообщения.
2. Создаётся `Dispatcher` с `MemoryStorage` — он принимает апдейты от Telegram и маршрутизирует их к нужным хэндлерам. `MemoryStorage` хранит состояния FSM в оперативной памяти (на следующих этапах заменим на Redis).
3. Регистрируется `DatabaseMiddleware` — чтобы каждый хэндлер автоматически получал сессию к БД.
4. Подключаются три роутера: `start`, `registration`, `profile`.
5. Запускается `dp.start_polling(bot)` — бесконечный цикл опроса Telegram на новые сообщения.

---

### `bot/db/models.py`

Описывает таблицы БД в виде Python-классов (ORM). SQLAlchemy превращает их в SQL-запросы автоматически.

**`User`** — строка в таблице `users`:
- `id` — внутренний первичный ключ
- `telegram_id` — уникальный идентификатор пользователя в Telegram (именно по нему мы узнаём, кто пишет боту)
- `created_at` — дата регистрации

**`UserProfile`** — строка в таблице `user_profiles`, связана с `User` один-к-одному:
- `name`, `bio` — имя и описание для анкеты
- `age`, `gender`, `seeking_gender` — основные поля для подбора
- `city`, `country` — география
- `age_min`, `age_max` — предпочтения по возрасту партнёра (заполняются на следующих этапах)

Между `User` и `UserProfile` настроен relationship — SQLAlchemy автоматически подгружает профиль при обращении к `user.profile`.

---

### `bot/db/session.py`

Создаёт два объекта, которые используются во всём приложении:

- `engine` — низкоуровневое соединение с PostgreSQL (пул соединений). `pool_pre_ping=True` означает: перед каждым использованием проверить, что соединение живо.
- `AsyncSessionFactory` — фабрика сессий. Каждый раз, когда нужно выполнить запрос к БД, вызываем `AsyncSessionFactory()` и получаем новую сессию. `expire_on_commit=False` — объекты не "протухают" после коммита, что важно для async-кода.

---

### `bot/middlewares/db.py`

**Middleware** в aiogram — это прослойка, которая выполняется до и/или после каждого хэндлера.

`DatabaseMiddleware` делает одно: открывает сессию к БД (`async with AsyncSessionFactory() as session`) и кладёт её в словарь `data["session"]`. После того как хэндлер отработал — сессия автоматически закрывается.

Благодаря этому любой хэндлер может просто объявить параметр `session: AsyncSession` и получить готовую сессию — не нужно открывать/закрывать её вручную в каждом месте.

---

### `bot/states/registration.py`

Определяет **состояния FSM** (Finite State Machine — конечный автомат) для процесса регистрации.

FSM позволяет вести "диалог" с пользователем по шагам: бот запоминает, на каком шаге находится конкретный пользователь, и применяет нужный хэндлер. Без FSM все сообщения обрабатывались бы одним и тем же кодом.

Состояния по порядку:
```
name → age → gender → seeking_gender → city → bio
```

---

### `bot/handlers/start.py`

Обрабатывает команду `/start`.

Логика:
1. Сбрасывает любое текущее FSM-состояние (`await state.clear()`) — на случай, если пользователь прервал регистрацию и начинает заново.
2. Ищет пользователя в БД по `telegram_id`.
3. **Если не найден** — приветствует и переводит в состояние `RegistrationStates.name` (начало регистрации).
4. **Если найден** — приветствует по имени из профиля и показывает главное меню.

---

### `bot/handlers/registration.py`

Самый большой хэндлер. Реализует пошаговую регистрацию через FSM.

Каждый шаг — отдельная функция с фильтром по состоянию:

| Функция | Состояние | Что происходит |
|---|---|---|
| `process_name` | `RegistrationStates.name` | Валидирует длину имени (2–50 символов), сохраняет в FSM-данные, переходит к возрасту |
| `process_age` | `RegistrationStates.age` | Проверяет что введено число от 16 до 100, переходит к полу |
| `process_gender` | `RegistrationStates.gender` | Обрабатывает нажатие инлайн-кнопки (`gender:male` / `gender:female`), переходит к поиску |
| `process_seeking` | `RegistrationStates.seeking_gender` | То же самое для предпочтений партнёра |
| `process_city` | `RegistrationStates.city` | Валидирует город, переходит к bio |
| `process_bio` / `skip_bio` | `RegistrationStates.bio` | Принимает текст или обрабатывает нажатие «Пропустить» |

После последнего шага вызывается `_save_and_finish`:
- Читает все собранные данные из FSM (`state.get_data()`)
- Создаёт запись `User` в БД, получает `user.id` через `flush()`
- Создаёт `UserProfile` с привязкой к этому `user.id`
- Делает `commit()` — данные записываются в PostgreSQL
- Показывает пользователю готовую анкету

---

### `bot/handlers/profile.py`

Обрабатывает кнопки главного меню.

**«👤 Моя анкета»** — загружает пользователя из БД вместе с профилем (через `selectinload` — один SQL-запрос с JOIN) и отображает анкету в читаемом виде.

**«✏️ Редактировать анкету»** — удаляет текущего пользователя (каскадно удаляется и профиль), после чего запускает FSM регистрации заново. Простейший способ "переписать" анкету на этапе 2.

---

### `bot/keyboards/reply.py`

**Reply-клавиатура** — кнопки, которые заменяют стандартную клавиатуру телефона. Появляются после завершения регистрации.

Содержит `main_menu_keyboard()` с двумя кнопками: «Моя анкета» и «Редактировать анкету».

---

### `bot/keyboards/inline.py`

**Inline-клавиатура** — кнопки, которые прикреплены к конкретному сообщению бота.

- `gender_keyboard()` — кнопки «Мужской» / «Женский» при выборе пола. При нажатии отправляется `callback_data` вида `gender:male`.
- `seeking_keyboard()` — кнопки выбора кого ищет пользователь: «Мужчину» / «Женщину» / «Не важно».
- `skip_keyboard()` — кнопка «Пропустить» на шаге bio.

Callback-данные (`gender:male`, `seeking:female`, `skip`) перехватываются в `registration.py` через фильтры `F.data.startswith(...)`.

---

### `alembic/versions/001_initial_schema.py`

Первая миграция — создаёт всю схему БД из нуля.

Команда `alembic upgrade head` применяет эту миграцию: выполняет функцию `upgrade()`, которая создаёт все таблицы в правильном порядке (сначала `users`, потом всё что ссылается на неё).

Команда `alembic downgrade base` откатывает всё назад через `downgrade()`.

Таблицы, которые создаются:
`users` → `user_profiles` → `interests` → `user_interests` → `user_photos` → `likes` → `matches` → `dialogs` → `messages` → `ratings` → `rating_events` → `referrals`

---

### `docker-compose.yml`

Поднимает два контейнера:

- **`postgres`** — база данных PostgreSQL 16. Данные хранятся в named volume `postgres_data`, так что перезапуск контейнера не удаляет их. Healthcheck проверяет готовность БД перед стартом бота.
- **`bot`** — сам бот. Собирается из `Dockerfile`. Ждёт пока postgres станет healthy, затем запускает `alembic upgrade head && python -m bot.main`.

---

### `Dockerfile`

Инструкция сборки Docker-образа бота:
1. Берём минимальный образ `python:3.12-slim`
2. Копируем `requirements.txt` и устанавливаем зависимости
3. Копируем весь код проекта
4. При старте контейнера: применяем миграции → запускаем бота
